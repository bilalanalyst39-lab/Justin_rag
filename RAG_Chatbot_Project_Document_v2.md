## 1. Project Overview

### 1.1 What is this Project?

A RAG (Retrieval-Augmented Generation) chatbot that:
- Ingests content from **RSS feeds** (podcasts) and **PDF documents** (CVs, reports, etc.)
- Stores processed content in a **Vector Database** for semantic search
- Retrieves relevant information based on user queries
- **Generates new content** (interview questions, episode briefs, summaries)
- Falls back to **web search** when knowledge base doesn't have the answer

### 1.2 Key Features

| Feature | Description |
|---------|-------------|
| **Multi-Source Ingestion** | RSS feeds (podcasts) + PDF documents |
| **Intelligent Deduplication** | Detects existing RSS feeds and only processes NEW episodes |
| **Disambiguation** | Handles multiple matches (e.g., two people with same name) |
| **Content Generation** | Creates interview questions, briefs, summaries from KB data |
| **Web Search Fallback** | Searches web when KB doesn't have answer |
| **Single Processing Tool** | Docling handles both audio AND documents |

#### 4.1.2 RSS Processing Logic Summary

| Step | Action | Purpose |
|------|--------|---------|
| 1 | Check if feed URL exists in database | Determine if new or existing feed |
| 2 | If exists: Get all processed episode GUIDs | Know what's already done |
| 3 | Parse current RSS feed | Get latest episode list |
| 4 | Compare GUIDs (current - existing) | Find NEW episodes only |
| 5 | Process only new episodes | Avoid redundancy |
| 6 | Update metadata database | Track what's processed |

#### 4.1.3 Episode Processing Steps

For each new episode:
1. **Download** - Fetch audio file from RSS enclosure URL
2. **Transcribe** - Use Docling ASR to convert audio to text
3. **Chunk** - Split transcript into smaller pieces (500 tokens, 50 overlap)
4. **Embed** - Generate vector embeddings for each chunk
5. **Store** - Save to Qdrant with rich metadata
6. **Track** - Mark episode as completed in PostgreSQL
